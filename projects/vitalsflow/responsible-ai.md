# Responsible AI Documentation — VitalsFlow

> CareDevi AI Innovation Hackathon 2026 | Track: AI Patient Triage

---

## 1. Project Overview

**VitalsFlow** is an AI-powered clinical decision support tool that analyzes real-time patient vitals against historical EHR data to generate triage risk scores using the NEWS2 protocol. It is designed to *assist* clinicians — not replace clinical judgment.

**Intended users:** Emergency department nurses and triage clinicians
**Intended use:** Supplementary triage decision support in clinical environments
**Not intended for:** Autonomous clinical decisions, diagnosis, or use without clinician oversight

## 2. Data Sources

### FHIR Data
| Source | Type | Status |
|---|---|---|
| HAPI FHIR Public Test Server (`hapi.fhir.org/baseR4`) | Synthetic patient data | ✅ Used in demo |
| Real EHR systems (Epic, Cerner, etc.) | Real patient data | ❌ Not used — production path only |

**Key statement:** This demo uses **100% synthetic patient data** generated by the HAPI FHIR public test server. No real patient data, personally identifiable information (PII), or protected health information (PHI) is collected, stored, or processed.

### Vitals Input
Vitals are entered manually by the clinician in the UI for demo purposes. In production, these would be pulled from bedside monitoring systems via FHIR `Observation` resources.

### Data Retention
- No patient data is persisted to any database in this demo.
- All data flows in-memory per API request and is discarded after the response.
- Gemini API calls may be subject to Google's standard data retention policies. In production, a BAA (Business Associate Agreement) with the LLM provider would be required.

## 3. Model Details

### AI Model
| Property | Value |
|---|---|
| Model | Google Gemini 1.5 Flash |
| Provider | Google AI Studio |
| Version | `gemini-1.5-flash` |
| Access | REST API with API key |
| Cost | Free tier (15 RPM) |

### Why Gemini 1.5 Flash
- Sufficient reasoning capability for structured clinical output
- Generous free tier suitable for hackathon prototype
- Low latency (~1–2s) appropriate for real-time triage context

### Clinical Protocol
The model is instructed to follow the **NEWS2 (National Early Warning Score 2)** protocol — a clinically validated, standardised early warning scoring system published by the Royal College of Physicians (UK). NEWS2 is used in NHS trusts and many international healthcare systems.

Using a named, validated protocol as the AI's reasoning framework:
- Reduces hallucination risk by anchoring output to established parameters
- Makes the AI's reasoning auditable ("the model was instructed to follow NEWS2")
- Aligns with clinical governance expectations

### Model Prompt Design
The system prompt instructs the model to act as a senior triage nurse applying NEWS2. It requires structured JSON output with a defined schema, enforced via Pydantic v2 output parsing. The `temperature` parameter is set to `0.1` to minimise variability in clinical reasoning output.

## 4. Bias Considerations

### Known Risks

**Training data bias**
Large language models are trained on general internet data and medical literature that may overrepresent certain populations (e.g., Western, English-speaking, male patients). NEWS2 itself has been validated across diverse populations, which mitigates some model bias by constraining reasoning to a standardised protocol.

**Vital sign interpretation bias**
Normal ranges for vitals can vary by age, sex, and baseline health status. The current implementation does not adjust NEWS2 thresholds for these factors. For example, a resting heart rate of 90 bpm means something different for a trained athlete vs. a sedentary elderly patient.

**Language and literacy bias**
The system prompt and output are in English only. In multilingual clinical settings, this limits accessibility.

**Socioeconomic bias**
Conditions like hypertension or diabetes — which affect NEWS2 scoring — disproportionately affect lower socioeconomic groups. The AI has no mechanism to contextualise these structural factors.

### Mitigation Measures Implemented
- All AI suggestions require explicit clinician Approve/Dismiss before any action
- Risk score is displayed alongside `justification` text — clinicians can read the AI's reasoning
- A fallback "Manual clinical assessment required" response triggers if the AI output is malformed
- No autonomous actions are taken without human confirmation

### Mitigation Measures for Production
- Regular bias audits across demographic subgroups
- Clinician feedback loop to flag incorrect risk scores
- Population-specific NEWS2 threshold adjustments (e.g., paediatric, obstetric)
- Multilingual prompt support

## 5. Failure Cases and Safety Handling

### Identified Failure Modes

| Failure Mode | Likelihood | Impact | Mitigation |
|---|---|---|---|
| LLM returns malformed JSON | Low–Medium | High | Pydantic parsing with safe fallback response |
| LLM hallucinates implausible risk score | Low | High | Clinician review required before any action |
| HAPI FHIR server unavailable | Medium | Medium | Loading skeleton UI; manual vitals still scoreable |
| Gemini API rate limit hit (15 RPM free tier) | Medium | Medium | Exponential backoff; Groq/Llama 3 as backup |
| Incorrect patient selected | Low | High | Patient name + DOB shown prominently before triage |
| Vitals entered incorrectly | Medium | High | Input validation with physiological range checks |
| LLM returns "I cannot assess" | Low | Medium | Fallback to manual review state; no score shown |

### Safety-Critical Design Decisions

1. **Human-in-the-loop is mandatory.** The Approve/Dismiss mechanism is not cosmetic — no FHIR `ServiceRequest` is written without explicit clinician action.

2. **AI output is labelled as AI-generated.** The UI clearly indicates that risk scores and suggested actions are AI-generated and require clinical verification.

3. **Graceful degradation.** If the AI layer fails entirely, the system still displays the patient's FHIR data and vitals — the clinician is never left with nothing.

4. **No storage of AI outputs.** Risk scores and suggested actions are not persisted, preventing stale AI recommendations from being acted upon later.

## 6. Transparency and Explainability

Every triage output includes a `justification` field — a 2–3 sentence plain-English explanation of why the AI assigned the score it did. This is displayed prominently in the Action Center alongside the score.

Example:
> "Patient presents with a respiratory rate of 22/min and SpO2 of 91%, both scoring 2 points on NEWS2. Combined with tachycardia (HR 98) and a temperature of 38.4°C, the total NEWS2 score of 7 indicates high clinical risk requiring urgent senior review."

This explainability is intentional — it allows the clinician to quickly verify whether the AI's reasoning matches their own assessment, and to override with confidence if not.

## 7. Regulatory Awareness

### Current Status (Hackathon Demo)
This prototype is a **research and development demo**. It is not a regulated medical device and has not been submitted for regulatory approval.

### Production Path
A production deployment of VitalsFlow would require:

| Requirement | Regulatory Body | Notes |
|---|---|---|
| Medical device classification | FDA (US) / MHRA (UK) / MDR (EU) | AI triage tools likely classified as SaMD (Software as a Medical Device) |
| HIPAA compliance | HHS (US) | BAA with all vendors, PHI encryption, access controls, audit logging |
| SMART-on-FHIR OAuth 2.0 | HL7 / ONC | Required for real EHR integration |
| Clinical validation study | IRB/Ethics Committee | Prospective validation of NEWS2 AI implementation |
| Audit trail | Clinical governance | Full logging of all AI recommendations and clinician actions |

### AI Act (EU) Considerations
Under the EU AI Act, a clinical triage tool would likely be classified as **high-risk AI** (Annex III, medical devices). This would require conformity assessment, technical documentation, and registration in the EU database before deployment.
